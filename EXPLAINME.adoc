The README makes claims. This file backs them up with domain-specific data structures and workflows.

== Claim 1: Workplace Mapping with Member Engagement Tracking and Spatial Analytics

From README:
____
Workplace Mapping: Detailed visualization of sites, departments, and shifts to identify organizing gaps. Member Engagement: Track the "Organizer Conversation" lifecycle and sentiment across the workforce.
____

**How it's proven:**
- **Worksite struct**: `Worksite` exported line 49; created via `register_worksite(employer, headcount)` (README line 37)
- **Member records**: `MemberRecord` exported line 49; linked to worksite with location metadata
- **Engagement tracking**: `OrganizerConversation` exported line 49; logged via `log_conversation(member_id, message, sentiment=0.8)` (README line 40, line 50 export)
- **Spatial location**: `GeoLocation` exported line 49; used in `find_members_near(lat, lon, radius)` (exported line 53)
- **Critical path**: `src/mapping.jl` (5KB) implements worksite structures, department hierarchy, shift allocation; `src/organizing.jl` logs organizer conversations with sentiment scores; `find_members_near` and `spatial_mashup` (exported line 53) perform geo-spatial queries
- **Sentiment analysis**: Conversation logging stores sentiment as float ∈ [-1, 1]; used for engagement heatmaps and progress tracking

**Caveat:** Spatial queries assume simple Euclidean distance (lat/lon). No geographic projection or boundary-aware clustering implemented. For large-scale organizing campaigns, integration with PostGIS or similar would be necessary for production use.

== Claim 2: Grievance Lifecycle Management with Immutable Case Logs and Collective Bargaining Support

From README:
____
Grievance Pipeline: Integrated intake, evidence logging, and deadline management for contract enforcement. Bargaining Support: Side-by-side contract clause comparison and fiscal costing of proposals.
____

**How it's proven:**
- **Grievance struct**: `GrievanceCase` exported line 49; created via `open_grievance(member_id, description)` (README line 43, line 50 export)
- **Case tracking**: `update_grievance_status(case, new_status)` (exported line 51) manages state transitions; evidence logging via case metadata
- **Contract clauses**: `ContractClause` exported line 49; retrieved and compared via `compare_clauses(clause1, clause2)` (exported line 52)
- **Bargaining proposals**: `BargainingProposal` exported line 49; created via `build_proposal(clause_id, description)` (README line 46)
- **Fiscal costing**: `cost_proposal(proposal)` (exported line 52) calculates total unit cost and economic impact
- **Critical path**: `src/grievances.jl` implements case intake and status management; `src/bargaining.jl` (575 bytes) manages clause comparison and proposal costing; immutability achieved via struct field freezing (no modification after creation, only status updates)
- **Example workflow**: README lines 43 shows case opening; line 46-47 shows proposal building and costing

**Caveat:** The grievance lifecycle has no built-in deadline enforcement or automatic escalation. Deadline management is currently data-only (stored in metadata). A production system would add email alerts, escalation workflows, and deadline-based state transitions.

== Dogfooded Across The Account

TradeUnionist.jl is the **labor analytics tier** for ecosystem operations:
- **PRComms.jl**: Uses TradeUnionist data for member alerts and union press releases (via `comms_bridge.jl` export at line 60)
- **PostDisciplinary.jl**: TradeUnionist labor economics data feeds into cross-disciplinary research (union density as `LinkedEntity`)
- **Cliodynamics.jl** (planned): Historical union membership trends and collective bargaining outcomes

Same domain-modeling pattern used in `idaptik` (game economics), `opsm` (package management), `prcomms` (stakeholder communication).

== Critical Invariants (What Must Not Break)

1. **Member data immutability**: Once a `MemberRecord` is stored, only status and engagement_level can be updated; core identity fields (name, ID) are frozen
2. **Grievance case idempotence**: Two calls to `open_grievance(same_member, same_description)` on same day must return same case ID (no duplicate filing)
3. **Metric consistency**: `calc_density`, `calc_coverage`, `calc_leadership_ratio` must all reference the same member set (transactions must not interleave)
4. **Spatial query correctness**: `find_members_near(lat, lon, r)` must be symmetric—if member A is near member B, then B is near A
5. **Costing accuracy**: `cost_proposal` must correctly aggregate all clause costs; no rounding errors >$0.01
6. **PR integration atomicity**: A member alert sent via PRComms must log to TradeUnionist audit trail atomically
